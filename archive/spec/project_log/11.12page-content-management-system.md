# Page Content Management System - Implementation Document

**Date:** November 12, 2025
**Feature:** Dynamic Page Content Management via Sanity CMS
**Status:** ✅ Complete

## Overview

Implemented a comprehensive content management system that allows the studio owner to edit all text content across the website through an admin dashboard interface, without needing to touch code. Content is stored in Sanity CMS and fetches with smart caching and default fallbacks.

## Problem Solved

Previously, all page text (hero headings, descriptions, event page content, etc.) was hardcoded in React components. Any text change required:
- Developer intervention
- Code deployment
- No flexibility for the business owner

## Solution

Created a tabbed admin interface at `/admin/dashboard/page-descriptions` where the owner can edit:

### Homepage Content:
- Hero section (heading + 2 CTA buttons)
- Main section (title + description)
- Offerings section (title, subtitle, 3 offering cards)
- Upcoming workshops section (label, title, subtitle)

### Event Pages:
- Adult Classes (title + description)
- Kid Classes (title + description)
- Camps (title + description)
- Private Events (title + description)

### Other Pages:
- Reservations (title + description)
- Gallery (title)
- About (title + description)

---

## Architecture

### Data Flow
```
Admin UI → API Route → Sanity CMS → Frontend Components
   ↓                                        ↓
 Edit text                            Display with fallback
```

### Key Design Decisions

1. **Single Document Model**: All page content stored in one Sanity document (`pageContent`) with nested structure
2. **Smart Caching**: Custom React hook caches content globally, only fetches once per page load
3. **Fallback Defaults**: Every field has hardcoded defaults, site never breaks if CMS is empty
4. **Tab-Based UI**: Organized by page type (Homepage/Event Pages/Other Pages) for clarity
5. **Simple Text Editing**: Currently using textarea (can upgrade to rich text later)

## Technical Implementation

### 1. Sanity Schema Structure
```typescript
pageContent (single document):
  ├── homepage
  │   ├── hero { heading, ctaButton1, ctaButton2 }
  │   ├── mainSection { title, description }
  │   ├── offerings { sectionTitle, sectionSubtitle, artCamps, classesWorkshops, privateEvents }
  │   └── upcomingWorkshops { label, title, subtitle }
  ├── eventPages
  │   ├── adultClasses { title, description }
  │   ├── kidClasses { title, description }
  │   ├── camps { title, description }
  │   └── privateEvents { title, description }
  └── otherPages
      ├── reservations { title, description }
      ├── gallery { title, description }
      └── about { title, description }
```

### 2. Custom Hook Pattern
```typescript
const { content, isLoading } = usePageContent();

// Usage in component
<h1>{content?.homepage?.hero?.heading || DEFAULT_TEXT.homepage.hero.heading}</h1>
```

**Benefits:**
- Single API call per page load
- Automatic caching across all components
- Type-safe access with optional chaining
- Always has fallback

### 3. API Endpoints

**GET `/api/page-content`**
- Fetches from Sanity CMS
- 60-second revalidation cache
- Returns: `{ success: true, data: PageContent }`

**PUT `/api/page-content`**
- Updates or creates Sanity document
- Patches existing document or creates new
- Returns updated document

## User Flow

### Admin Side:
1. Navigate to `/admin/dashboard/page-descriptions`
2. Select tab (Homepage/Event Pages/Other Pages)
3. Edit text in input fields and textareas
4. Click "Save All Content"
5. Wait ~1 minute for Sanity cache to update

### Frontend Side:
1. Components call `usePageContent()` hook
2. Hook fetches from API (once, then cached)
3. Components render with CMS content
4. If no CMS content exists, displays defaults
5. User sees updated text after cache refresh

## Performance Considerations

### Optimizations Implemented:
1. **Global Caching**: Single fetch shared across all components
2. **Lazy Evaluation**: Only fetches when first component mounts
3. **Promise Deduplication**: Multiple concurrent requests share same promise
4. **60s API Cache**: Sanity results cached at API level
5. **Default Fallbacks**: No blocking if CMS unavailable

### Metrics:
- **API Calls per page**: 1 (down from potential 11+)
- **Cache Duration**: Until page reload
- **Render Performance**: No impact (cached content)

## Known Limitations

1. **Text Only**: Currently only supports plain text, no formatting
2. **No Media**: Cannot upload/manage images through this interface
3. **Cache Delay**: Changes take ~1 minute to appear (60s revalidation)
4. **Single Language**: No translation support
5. **No Versioning**: Cannot undo or see change history

## Maintenance Notes

### To Add New Editable Content:
1. Update Sanity schema in `pageContentType.ts`
2. Add TypeScript types to `types/pageContent.ts`
3. Add defaults to `lib/constants/defaultPageContent.ts`
4. Add form fields to admin page
5. Update component to use `usePageContent()` hook

### To Clear Cache:
```typescript
import { clearPageContentCache } from "@/hooks/usePageContent";
clearPageContentCache(); // Call after admin saves
```
